## 2. Analysis

### Objective

Prevent public commercial routes from being treated as generic page design work and define them instead as governed trust, proof, interaction, and conversion systems.

### Scope

This amendment applies to:

- homepage systems
- packages and pricing-decision pages
- case-study routes and proof-heavy narrative pages
- service pages
- shared route behaviors that materially affect trust, proof, readability, motion, and CTA logic

### Required Elements

- route-level visual hierarchy ownership
- proof-presentation control
- interaction and motion discipline
- readability and fatigue review
- CTA-path clarity
- accessibility and reduced-motion checks
- SEO and semantic preservation when route structure changes
- QAS release control and Founder human gate where market-facing posture changes materially

### Exclusions

- net-new pricing strategy
- fabrication of metrics, logos, testimonials, or outcomes
- unapproved brand identity redesign
- legal approval authority
- unauthorized deployment or publication
- unapproved scope expansion beyond the named public-route classes

### Canonical Position

`VECS` is approved as a permanent architectural department in the NoDrftSystems operating model.

For current operations, `VECS` functions as a virtual governed overlay executed through existing approved roles rather than as a new standalone staffing bench. This is the optimized path because it preserves the real value of the old concept without duplicating roles that already exist in the live architecture.

This amendment therefore changes governance and activation logic now, while deferring any separate registry/skill-pack expansion until there is evidence that the current bench cannot cover the workload with sufficient precision.

### VECS Role Mapping To The Current Bench

| VECS responsibility | Current approved role(s) |
| --- | --- |
| Route architecture, authority flow, proof rhythm | `VDA` |
| Treatment planning, section translation, layout guidance | `DAA` |
| Executable frontend implementation | `FIS` |
| Anti-generic posture and pattern rejection | `BCA` |

| Accessibility and reduced-motion compliance | `AAA` |
| Plain-language clarity and CTA comprehension | `PLA` |
| Bilingual parity after structural change | `BPA` and `TCA` when needed |
| Proof verification and evidence packaging | `reviewer_public_proof` plus `QDA` |
| Release hold / release approval inside workflow | `QAS` |

### VECS Activation Rule

The VECS overlay is mandatory whenever a governed build materially changes:

- homepage hierarchy or authority flow
- packages-page comparison logic or package framing
- case-study proof presentation, before/after structure, or metric storytelling
- service-page trust flow, process explanation, FAQ framing, or CTA-path logic

No such route change may be treated as "frontend only" work.

### Hard Rules

- No fabricated metrics, logos, testimonials, client results, before/after evidence, or outcomes.
- No motion-dependent comprehension. Reduced-motion handling is mandatory.
- No generic, templated, visually flat, or commercially weak route posture on governed public routes.
- No route release without proof review, readability review, accessibility review, and QAS sign-off.
- Founder review is mandatory when route changes materially alter package presentation, public trust posture, or market-facing proof framing.

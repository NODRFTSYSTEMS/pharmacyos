# SOP Inventory

# SOP-001: Intake
Purpose: Keep funnels separate (Inquiries vs Start an Engagement)
Owner: Growth/Qualification Lead
Trigger: New inquiry received
Outputs: Routed inquiry + response template
## Overview
This SOP ensures inquiries are properly categorized and routed to the appropriate workflow. The key distinction is between project intake (Start an Engagement) and general inquiries.
## Steps
Receive inquiry via website form, email, or referral
Categorize: Project inquiry vs General inquiry vs Partnership
For project inquiries: Route to Start an Engagement form
For general inquiries: Log and respond within 24 hours
For partnerships: Flag for Founder review
## Decision Tree
Does the inquiry mention a specific project, timeline, or budget? → Route to Start an Engagement. Is it general contact, media, or partnership? → General inquiry path. Is it unclear? → Respond asking for clarification.

# SOP-002: Qualification
Purpose: Fit screening + package routing
Owner: Growth Lead + Founder
Trigger: Completed Start an Engagement form
Outputs: Recommended package + next step
## Overview
This SOP ensures every prospect is evaluated against fit criteria before proposal. The Client Evaluation Scorecard is mandatory.
## Steps
Review completed Start an Engagement form
Score against Client Evaluation Scorecard (minimum 75/100)
Identify red flags (vague scope, unallocated budget, >3 decision makers)
Determine recommended package based on scope
Apply risk pricing multipliers if needed
Prepare proposal or decline with referral
## Decision Matrix

# SOP-003: Discovery Sprint
Purpose: Clarify scope before build
Owner: Founder + UX/Content Lead
Trigger: Discovery Sprint contracted
Outputs: Scope brief + package recommendation
## Overview
This SOP governs the Discovery Sprint delivery to ensure scope clarity before build commitment.
## Steps
Schedule 2-hour discovery workshop with client
Pre-workshop: Send questionnaire (business objectives, constraints, stakeholders)
Workshop: Facilitate structured discussion covering objectives, users, content, technical requirements
Document: Create scope brief within 48 hours
Review: Present scope brief to client for validation
Recommend: Propose appropriate package and timeline
## Deliverables
Scope brief document (business context, objectives, requirements, constraints)
Execution plan (phases, timeline, dependencies)
Package recommendation with pricing

# SOP-004: Build Execution
Purpose: Bounded delivery with defined outputs
Owner: Implementation Lead
Trigger: Contract signed, deposit received
Outputs: Build plan + deliverables checklist
## Overview
This SOP governs the build phase to ensure on-scope, on-time delivery with quality outcomes.
## Steps
Kickoff: Schedule client kickoff call, confirm scope and timeline
Plan: Create detailed build plan with milestones
Build: Execute with AI-augmented production, ARE validation at each milestone
Review: Present work at each milestone for client feedback
Revise: One round of revisions included; additional billed separately
Finalize: Complete all deliverables per checklist
## Milestone Structure
M1: Design approval (wireframes/mockups)
M2: Development checkpoint (50% complete)
M3: Content integration
M4: QA and refinement
M5: Final approval and handoff

# SOP-005: QA Multi-Pass
Purpose: Package integrity + language + pricing + proof
Owner: ARE + specialist reviewers
Trigger: Build complete, ready for review
Outputs: QA checklist + fix list
## Overview
This SOP ensures comprehensive quality review across multiple dimensions before release.
## Review Passes
### 1. Package Integrity Review
Ladder matches commercial offer
Inclusion/exclusion frames present
Upgrade logic shown
'From' used only with real variability
### 2. Plain Language Review
No acronyms dropped cold
Glossary covers key terms
Buyer-safe explanations on-page
### 3. Pricing Safety Review
Only approved figures appear
Exclusions visible
No bait pricing
### 4. Public-Proof Review
Controlled snapshots
Confidentiality safe
No client names unless approved
### 5. Bilingual Review
Meaning, tone, CTA strength parity
Transcreation quality
### 6. Accessibility Review
Headings, labels, forms
WCAG 2.2 baseline compliance

# SOP-006: Release
Purpose: Human sign-off release discipline
Owner: ARE + Founder
Trigger: All QA passes complete
Outputs: Completion report + sign-off record
## Overview
This SOP ensures fail-closed release gates are passed before any public deployment.
## Release Gates

## Steps
ARE completes QA checklist and fix list
ARE generates completion report
Founder reviews completion report
Founder approves release (PROCEED) or requests changes (HOLD)
If PROCEED: Deploy to production
If HOLD: Return to build with specific fix requirements

# SOP-007: Careers
Purpose: Continuous but controlled talent intake
Owner: Growth Lead
Trigger: Application received
Outputs: Triage result + applicant record
## Overview
This SOP ensures careers remains live continuously while maintaining selectivity and privacy.
## Steps
Receive application via careers form
Acknowledge receipt within 48 hours
Categorize: Active role applicant vs Future opportunity vs Specialist interest
For active roles: Review against job requirements, schedule interview if qualified
For future opportunities: Add to talent pool, contact when role opens
For specialists: Evaluate portfolio/experience, add to vetted bench
Maintain applicant records for 24 months

# SOP-008: Contractor Onboarding
Purpose: Add specialists without losing control
Owner: Founder + ARE
Trigger: Specialist identified and vetted
Outputs: SOW + access controls + QA rules
## Overview
This SOP ensures specialists are onboarded with clear boundaries and quality controls.
## Steps
Identify need for specialist expertise
Source from vetted bench or new referral
Review portfolio/work samples
Execute professional services contract (not employment)
Define SOW: scope, deliverables, timeline, payment terms
Set up access controls (limited to specific project)
Communicate QA rules and review gates
ARE validates all deliverables before client delivery
## Contract Requirements
Multiple clients clause (contractor free to serve others)
Deliverables-based payment (not hourly/salary)
No exclusivity requirement
Contractor provides own equipment
No fixed hours (results-oriented)
